# Stream E Part 2: β Parameter Inconsistency — Diagnosis

**Date:** 2026-04-22
**Goal:** Resolve open Task #9 — "BETA parameter inconsistency (1.0 vs 3.0)"

---

## Two Different β's Exist

### Paper's β (the claim)

**Line 502–504 of `main.tex`:**

```
RIF_i = 1 - e^{-β · CKDI_i²}
with β = 3.0, encoding the assumption that recovery
effort grows faster than drift magnitude.
```

This is an **exponential saturation** curve. At CKDI = 1, RIF ≈ 0.95.

### Code's BETA (the actual implementation)

**`pipeline_ckdi_bdri.py` line 27:**

```python
BETA = 1.0   # mttr slope
```

**Line 174 & 178:**

```python
MTTR = MTTR_BASE + BETA * CKDI
RIF  = (MTTR - MTTR_BASE) / MTTR_BASE   # = BETA * CKDI
```

This is **linear**. At CKDI = 1, RIF = 1.0 (not 0.95).

---

## Problem

The two formulas are mathematically different:

| CKDI | Paper RIF (β=3.0) | Code RIF (BETA=1.0) |
|------|-------------------|---------------------|
| 0.0  | 0.000             | 0.000               |
| 0.25 | 0.173             | 0.250               |
| 0.50 | 0.528             | 0.500               |
| 0.75 | 0.816             | 0.750               |
| 1.00 | 0.950             | 1.000               |

The code does **not** reproduce the exponential-saturation RIF the paper claims.

Because the paper's Table III metrics and §V-G sensitivity results are all generated by code, the reported BDRI numbers are built on the **linear** RIF, even though the paper formally declares the exponential one.

---

## Recommended Fix (per standing rule "code must support paper claims")

Update `pipeline_ckdi_bdri.py` to implement the paper's RIF formula directly:

```python
# Replace lines 25-27:
DELTA = 1.0   # latency slope (paper δ)
ALPHA = 2.0   # availability decay (paper α)
BETA  = 3.0   # RIF exponential-saturation rate (paper β)

# Replace lines 172-178:
Latency = L_BASE * (1 + DELTA * CKDI)
Availability = np.exp(-ALPHA * CKDI)
RIF = 1.0 - np.exp(-BETA * CKDI**2)         # paper eq. line 502
# MTTR is not used in the paper's BDRI chain; retain for plotting only
MTTR = MTTR_BASE + CKDI                      # cosmetic, unused in BDRI
```

Then re-run the pipeline and regenerate Table III / Table IV.

**Expected effect on reported numbers:**

Since code's linear RIF over-weights high CKDI samples relative to paper's saturating RIF, the *optimized BDRI weights* will shift and Mean BDRI_atk values in Table IV will change. The qualitative claim ("strongly negative CKDI–BDRI correlation, r < -0.93") is robust and should still hold — the sensitivity section explicitly argues this.

---

## Option B (not recommended)

Change the paper to declare the linear RIF that the code actually uses. This is less defensible because (a) the paper's justification ("recovery effort grows faster than drift magnitude") specifically motivates the nonlinear form, and (b) β=3.0 is referenced in §IV and §V-G.

## Nomenclature Clarification (add to paper appendix or code docstrings)

There are **three distinct parameters named α/δ/β in this project**:

| Symbol   | Where               | Default | Meaning                                         |
|----------|---------------------|---------|-------------------------------------------------|
| α (CKDI) | `src/ckdi.py`       | 0.5     | CKDI composition weight (stat vs PCA)          |
| α (BDRI) | `pipeline_ckdi_bdri.py` `ALPHA` | 2.0 | Availability decay rate (paper eq. 479)       |
| δ        | `pipeline_ckdi_bdri.py` `DELTA` | 1.0 | Latency slope (paper eq. 478)                 |
| β        | `pipeline_ckdi_bdri.py` `BETA`  | 3.0 **(after fix)** | RIF saturation rate (paper eq. 502) |

The CKDI-α and BDRI-α collision is a latent source of confusion. Recommend renaming CKDI-α → `stat_weight` in a follow-up refactor.
